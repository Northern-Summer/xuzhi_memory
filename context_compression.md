# 上下文压缩机制

> 目的：减少每次 session 的上下文加载量
> 创建时间：2026-04-01 18:05
> 维护者：Ξ

---

## 问题分析

### 当前状态

```
Session 启动时加载：
- MEMORY.md: ~2k 行
- AGENTS.md: ~500 行
- TOOLS.md: ~250 行
- expert_tracker/: ~多个文件
- 注入记忆: ~1k 行

总计：~82k tokens
```

**问题**：
- 82k tokens 远超实际需要的上下文
- 大部分内容在当前 session 用不到
- 每次重复加载，Token 浪费严重

---

## 压缩策略

### 策略 1：分层加载

```
Layer 0（必须加载）：
  - 当前 session 任务
  - 最近断点（~50 行）
  - 红线规则
  
Layer 1（按需加载）：
  - MEMORY.md 核心部分（身份、架构）
  - AGENTS.md 启动规程
  
Layer 2（延迟加载）：
  - 详细工具文档
  - 历史记录
  - 框架学习笔记

Layer 3（不加载）：
  - 归档文件
  - 超过 30 天的记录
```

### 策略 2：增量更新

```
Session N 启动时：
  读取 LATEST.md（最近断点，~50 行）
  
Session N 结束时：
  写入断点摘要（~50 行）
  
Token 节省：82k → 5k（节省 94%）
```

### 策略 3：Compact 格式

```
完整版：
  - 详细解释
  - 示例代码
  - 历史背景
  
Compact 版：
  - 核心要点（3-5 条）
  - 关键路径
  - 引用链接（按需展开）
```

**行数**：~20 行（压缩 90%）

### Phase 2：分层加载（本周）

修改 AGENTS.md Session Startup：

```markdown
0. 读取断点摘要（~50 行）
1. 识别当前任务类型
2. 按需加载相关文档
3. 其他文档延迟加载
```

### Phase 3：智能预加载（下周）

```
分析历史 session：
  - 哪些文件经常一起使用？
  - 哪些文件很少使用？
  
建立预加载模型：
  - 任务类型 → 需要加载的文件
  - 减少不必要的加载
```
