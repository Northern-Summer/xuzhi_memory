# MEMORY_HYGIENE_RULES.md — 记忆卫生守则（强制性）

> **强制等级：最高。所有 agent 必须遵守。**
> **最后更新：2026-03-28**
> **参考框架：CrewAI Memory（复合评分）+ Zettelkasten（原子笔记）+ CoALA Memory Taxonomy（STM/LTM分类）**

---

## 记忆三层架构（强制）

```
L1 — Session 工作记忆
位置：~/.xuzhi_memory/memory/YYYY-MM-DD.md
定义：当天所有 session 的对话、任务、发现的原始记录（一天可能有多个 session）
保留期：7天 → 归档或删除
性质：所有 agent 共享，可读写
注意：粒度是 session，不是 day。一天可能有多个 session，每个 session 结束时应写交接备忘录

L2 — 学科私有记忆
位置：~/.xuzhi_memory/agents/{X}/memory/
定义：该 agent 学科特有的知识、笔记、分析
保留期：无期限
性质：agent 私有，不共享，不污染 L1/L3

L3 — 永久共同记忆
位置：~/.xuzhi_memory/memory/knowledge/
定义：跨学科重要事实、决策、原则
保留期：永久
性质：所有 agent 可读，L3 内容只能晋升，不能降级
晋升条件（必须同时满足）：
  1. 被至少2个 agent 独立验证
  2. 对系统运行有可量化影响
  3. 不依赖具体日期/事件
```

## 整理时机（强制，非可选）

### 连续保存原则（最高优先级，2026-03-28 确立）

**核心原则：每个原子操作后立即保存，不等"工作告一段落"。**

```
原子操作定义：
  - 完成一个文件写入（任何大小）
  - 完成一次网络请求
  - 完成一次 git commit
  - 完成一次 cron 触发
  - 任何超过 5 分钟的思考/执行过程

不等的情况：
  ❌ "等写完这一节再保存"
  ❌ "等这个任务完成再保存"
  ❌ "等 session 快结束时再统一保存"

正确做法：
  ✅ 写完一段代码 → 立即 commit
  ✅ 跑完一个脚本 → 立即写 log
  ✅ 完成一次分析 → 立即写 memory
  ✅ 每次 compact 前 → 必须有 flush
```

**断电/崩溃场景：**
- 两次保存间隔最大 5 分钟（人的极限等待）
- 任何超过 5 分钟的连续工作，必须在第 5 分钟时强制保存
- 不需要完整结论，只需要"做到了哪一步"

**Session compact 前：**
- compact 是 session 结束的唯一可信触发点
- 触发 compact 时，立即将所有未保存状态 flush 到 L1
- compact 前的 flush 不依赖"工作是否完成"
- **【2026-03-30 新增】技能提取（MetaClaw 机制融合）**：
  1. 检查本次会话是否有失败（错误、用户纠正、任务未完成）
  2. 有失败 → 分析失败原因 → 提取可迁移技能 → 写入 MEMORY.md 技能库
  3. 标记本会话为"support 数据"（已用于技能演化，不用于其他学习）
  4. 更新技能代数 g → g+1

## Session 启动整理

**操作**：
  1. 读当日 memory 文件，找到上一个 session 的交接备忘录
  2. 标记：哪些值得晋升 L3，哪些归档，哪些删除
  3. 更新上一个 session 的记录（用 --- 分隔）
  4. 删除超过7天的 L1 文件（用 gio trash，不用 rm）

## Session 结束整理

**操作**：
  1. 写当前 session 摘要（3-5条最重要的）
  2. 检查本次 session 有无需要晋升 L3 的内容
  3. 确认交接备忘录已写入当日 memory 文件

**交接备忘录格式**：
```
---

## Session交接备忘录 | 启动: YYYY-MM-DD HH:MM | 结束: YYYY-MM-DD HH:MM

### 完成项
- ✅ ...

### 未完成项
- 🟡 ...

### 系统状态
- Gateway: ...
- Expert Tracker: ...
- Cron: ...
- 今日当值: ...

### 核心教训（如有）
...

---
```

### 每周一整理（强制）
```
操作：
  1. 遍历上周所有 L1 文件
  2. 做复合评分（见下）
  3. 将高分内容晋升 L3
  4. 将低分/过时内容归档或删除
  5. 更新 L3 索引文件
```

### 每月末整理（强制）
```
操作：
  1. 遍历当月所有 L2 文件
  2. 检查学科知识是否有冲突或重复
  3. 清理无效链接、孤立文件
  4. 更新各 agent 的知识索引
```

## 复合评分标准（强制计算）

每个记忆条目评分：

```
分数 = (重要性 × 0.4) + (近期度 × 0.3) + (通用性 × 0.3)

重要性：
  5分 — 决策、原则、系统架构
  3分 — 方法、流程、工具
  1分 — 日常记录

近期度（半衰期14天）：
  5分 — 3天内
  4分 — 7天内
  3分 — 14天内
  1分 — 30天以上

通用性：
  5分 — 跨学科，任何 agent 都有用
  3分 — 多学科相关
  1分 — 仅本学科

评分 ≥4.0 → 考虑晋升 L3
评分 ≤1.5 → 标记删除
```

## 原子笔记标准（强制）

**每个 L2/L3 笔记必须满足：**

```
1. 一笔记一事实（原子性）
   ✅ "多智能体系统中，通信开销是主要瓶颈"
   ❌ "多智能体系统有很多问题，通信开销是主要瓶颈，还有信任问题"

2. 标题即索引（可搜索）
   ✅ "# 通信开销是多智能体协作的主要瓶颈"
   ❌ "# 多智能体的问题"

3. 包含链接（连接而非堆积）
   每条笔记必须链接到至少一条相关笔记
   使用 [[filename]] 格式

4. 有来源（可验证）
   包含：日期、来源、验证状态
```

## 跨 agent 隔离规则（强制）

```
1. L1 是共享的，但每个 agent 只写自己的 section
2. L2 是私有的，绝对不写入他人 L2 目录
3. L3 是共享的，但晋升需要理由（防止霸权）
4. handover 备忘录是 L1，但只写交出者的视角
5. 禁止：复制他人 L2 内容到自己的 L2（引用除外）
6. 禁止：未经共识修改他人留在 L1 的内容
```

## 违规检测（自动）

```python
# 每次 session 启动时自动检查
检查项：
  1. L1 文件是否有超过7天未处理的
  2. L2 文件是否有原子性违规
  3. 是否有 agent 写入了他人 L2 目录
  4. L3 晋升是否满足3个条件
  5. handover 备忘录是否完整
```

## 自动执行机制（强制，2026-03-28 补充）

### cron 层检查点（每小时自动）
```
每小时整点：检查今日 L1 是否已写入
- L1 文件不存在或 < 200字节 → 立即写入当前状态
- 不通知，不询问，不等待
```

### session compact 前强制 flush
```
触发时机：session 即将 compact 时
操作：立即将当前 session 所有未保存状态写入 L1
- 关键信息：任务进展、open questions、决策点
- 标记：[compact-flush] 前缀
```

### cron 层自动写入模板
```
~/.openclaw/workspace/memory_checkpoint.sh
- 整点运行
- 检查 ~/.xuzhi_memory/memory/YYYY-MM-DD.md
- 字节数 < 200 → 追加系统状态（cron job 名称、时间戳、最后活动）
```

### 违规自动报警
```
检测到违规 → 写入 ~/.xuzhi_memory/memory/hygiene_violations.log
- 不发通知（避免通知风暴）
- 不暂停（规则是纪律，不是惩罚）
- 可由 human 随时查询
```

### 核心原则（2026-03-28 确立）
```
规则是最低要求，不是上限。
执行是默认行为，不是选项。
违规 → 自动修复 + log，不等待指令。
```
