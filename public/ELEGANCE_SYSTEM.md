# 优雅原则 — Xuzhi 系统品味标准

> 来源：用户提供，2026-03-25 整理
> 位置：~/xuzhi_genesis/public/ELEGANCE_SYSTEM.md
> 性质：宪法级标准，服务于记忆与安全两大原则

---

## 核心主张

**优雅是超越功能正确性的更高追求。**

代码像一首诗，结构如一座精妙的建筑，每个模块各司其职，不冗余、不造作，让后来者读之如沐春风。

**优雅的检验标准：**
- 新人能否在无额外解释下快速理解代码？
- 修改一处功能时，影响范围是否可预测？
- 删除冗余代码时，是否无需担心破坏其他逻辑？
- 阅读代码时，是否有"美"的体验？

## 九维度优雅实践

### 一、命名：精确与韵律
- 语义明确：`getUserById` 优于 `fetch`
- 布尔用 `is/has/can` 前缀
- 领域驱动：业务术语而非技术术语
- 好命名 = 可执行的需求文档

### 二、代码结构：层次分明
- **SRP**：每个函数只做一件事，只有一个变化原因
- 函数不超过 20 行，嵌套不超过 2 层
- **卫语句**（Guard Clause）：提前 return 隔离异常
- **组合优于继承**：灵活可测试

### 三、抽象层次：恰到好处
- 同一层级的代码处于同一抽象层次
- **YAGNI**：不为"未来可能"引入不必要的接口
- **DIP**：高层不依赖低层，二者依赖抽象

### 四、一致性：系统之美
- 代码风格自动化（Black/Prettier/gofmt）
- 同一类问题用相同设计模式
- 错误处理统一：类型、码、包装方式一致

### 五、可读性：为读者而写
- **代码即注释**：命名和结构代替注释
- **显式优于隐式**：无魔法值、无全局隐式状态
- 合理换行与分组，视觉舒适

### 六、测试：优雅的防护网
- **测试即文档**：`should_do_something_when_condition` 命名
- **单一断言**：每个测试只验证一个行为
- 测试数据构建器：Builder 模式
- 覆盖率质量 > 数量

### 七、架构：整洁的边界
- 清晰分层，依赖方向向内
- 模块化按功能/领域划分，无循环依赖
- 按业务边界而非技术边界划分服务

### 八、自动化与工具链
- **Git 钩子**：提交前自动 lint/format/test
- **CI 流水线**：每次提交自动构建、测试、扫描
- **一键本地环境**：`docker-compose up` 拉起所有依赖

### 九、文档与沟通
- **自文档化**：命名、类型、包结构传递信息
- **ADR**：重要技术决策记录背景、选项、决策、后果
- 接口文档自动化（OpenAPI/Swagger）

## 优雅的本质：克制与匠心

简洁、一致、可维护、可读——用最朴素的方式解决问题，让代码像诗歌一样传达意图。

**优雅是一种文化：**
- 代码审查时讨论"是否优雅"，而不只是正确性
- 重构不只是加功能，而是消除"异味"
- 工具链把人从重复劳动中解放，专注于创造性表达

## 与宪法连接

### 记忆原则 → 优雅
- 记忆即存在 = 优雅的命名 = 代码自文档化
- 记忆的各层（L1/L2/L3）= 架构的分层
- 记忆不可删除 = 依赖倒置：高层不依赖低层

### 安全原则 → 优雅
- 安全验收 = 一致性（所有操作有统一检查）
- 备份 = 可观测性（系统状态可追踪）
- 混沌工程 = 自动化测试（故障注入自动化）

### 优雅本身也要优雅化
- 本文件语言简洁，无冗余
- 每条原则不超过三行
- 层次分明：核心主张 → 九维度 → 连接宪法
