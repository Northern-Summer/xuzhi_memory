## 二、Bootstrap链Spec（最优先）

### 2.1 问题本质

**当前状态（审计结果）：**
```
每次OpenClaw session启动：
  → OpenClaw从 ~/.openclaw/workspace-xi/ 注入文件到session context
  → workspace-xi/SOUL.md 是"污染版本"（含错误Rho）
  → workspace-xi/IDENTITY.md 可能是旧版本
  → session读到的SOUL与 ~/.xuzhi_memory/agents/xi/SOUL.md 不同
```

**根本问题：** OpenClaw的bootstrap注入是session级别的。workspace-xi/下的文件是"源"，每次新session都会重新注入。一次性覆盖workspace-xi/SOUL.md只管一次，下一个session又会被覆盖。

### 2.2 修复方案（拒绝补丁思维）

**错误的修复：** 每次session启动后同步SOUL文件。
**正确的修复：** 让workspace-xi/的文件本来就正确。

**方法：workspace-xi/作为symlink**

```
现状：
  ~/.openclaw/workspace-xi/ → 物理目录，内容来自session注入

目标：
  ~/.openclaw/workspace-xi/ → symlink → ~/.xuzhi_memory/agents/xi/workspace/
```

**但：** OpenClaw可能不支持symlink作为workspace。需验证。

**备选方案：workspace-xi/内容与xuzhi_memory保持同步**

```
如果OpenClaw强制使用物理目录：
  → 创建一个cron job或watchdog，在每次Xi写入SOUL/IDENTITY时，同步到workspace-xi/
  → 或者在Xi的session启动流程中，第一件事就是从xuzhi_memory读取并覆盖
```

### 2.3 Bootstrap Spec目标

| 属性 | 规范值 |
|------|--------|
| SOUL源 | `~/.xuzhi_memory/agents/xi/SOUL.md`（唯一真实数据源） |
| IDENTITY源 | `~/.xuzhi_memory/agents/xi/IDENTITY.md` |
| workspace-xi/ | **首选：symlink到xuzhi_memory/agents/xi/workspace/** |
| Bootstrap触发 | session启动时自动从workspace-xi/读取 |

**行动：**
- [ ] 验证OpenClaw是否支持workspace目录为symlink（测试：创建symlink，观察是否跟随）
- [ ] 如果支持：执行symlink方案
- [ ] 如果不支持：实现启动时同步脚本（每次Xi session第一行执行同步）

## 五、命名规范Spec（高优先级）

### 5.1 当前问题

```
ratings.json:          Ξ, Φ, Δ, Θ, Γ, Ω, Ψ（希腊）
OpenClaw注册:          phi, delta, theta, gamma, omega, psi, rho（拉丁）
OpenClaw目录:          xi/, Ξ/, Ρ/（混合）
代码引用:              'phi', 'delta'（拉丁字符串）
generate_agent_name.py: "rho": "ρ"（小写希腊rho，应为"Rho": "Ρ"）
```

### 5.2 Spec决策

**决策：拉丁小写为内部标准。**

| 用途 | 标准 |
|------|------|
| OpenClaw注册文件名 | `{code}.json`（小写拉丁：phi.json, delta.json...） |
| OpenClaw agent目录 | `{code}/`（小写拉丁：xi/, phi/, delta/...） |
| 代码/变量引用 | `'phi'`, `'delta'`（拉丁字符串） |
| ratings.json | **维持希腊字母**（面向人类可读，不改） |
| 会话显示名 | 维持希腊（ΞΦΔΘΓΩΨ） |

**理由：**
- 改ratings.json会影响所有评分系统（高风险）
- 改OpenClaw注册会改变session路由（高风险）
- 改代码引用需要全面搜索替换（高风险）
- 拉丁小写是代码层标准，希腊是展示层标准，两者共存但服务于不同用途

**不改变：**
- ratings.json的希腊字母（展示层）
- 会话label的希腊字母（展示层）

**要改变：**
- generate_agent_name.py的rho→ρ改为rho→ρ（小写拉丁统一）
- Ξ/目录名统一为xi/（如果文件系统区分大小写）
- Ρ/目录删除或重命名

**行动：**
- [ ] 搜索所有代码引用，确认Greek vs Latin的使用场景
- [ ] 修改generate_agent_name.py中的大小写
- [ ] 统一目录名为小写拉丁

| Gate 2 | Inventory完成 | 全量文件清点报告已输出，人类确认 |
| Gate 3 | 决策完成 | 每个文件的三类决策已输出，人类审批 |
| Gate 4 | P0修复完成 | T6-T10全部通过 |
| Gate 5 | P1修复完成 | T1-T5,T11-T13全部通过 |
| Gate 6 | P2修复完成 | T14-T16全部通过 |
| Gate 7 | 最终验收 | 全部16项测试通过 |

**每个Gate：必须输出书面报告 → 人类审批 → 才能进入下一阶段。**

## 十、Emergency Stop机制

**触发条件（满足任一）：**
- Pre-Check任意项失败
- Migration过程中Gateway崩溃
- 连续3个文件操作失败
- git操作出现无法自动解决的冲突
- 任意阶段完成后的human审批拒绝

**Stop后的操作：**
1. 记录当前进度到修复日志
2. 不继续任何写操作
3. 报告人类，等待决策

**Rollback范围：**
| 已完成阶段 | Rollback方法 |
|-----------|-------------|
| Phase 1 (Inventory) | 无需Rollback（只读） |
| Phase 2 (Decision) | 无需Rollback（只是决策） |
| Phase 3.1 (P0修复) | `gio trash`的文件从备份恢复；代码修改用git |
| Phase 3.2 (P1修复) | git checkout（但删除操作的文件需从备份恢复） |
| Phase 3.3 (P2修复) | 停止cron/systemd后用备份恢复 |
| Phase 3.4 (清理) | 确认备份有效性后，可永久删除 |
